# Hunter AI — Integration API Spec

**Версия 1.0**

Документ для разработчиков CRM-системы, которая хочет подключиться к Hunter AI.
Читать внутреннее устройство Hunter AI не нужно — здесь описан **единый формат данных**,
который ваша сторона должна отдавать. Реализуете его один раз («мостик») — и всё работает.

---

## 1. Что такое Hunter AI и зачем эта интеграция

Hunter AI — это аналитический слой поверх вашей CRM. После подключения он читает данные о
продажах (лиды, звонки, результаты) и превращает их в живые дашборды, показатели по каждому
менеджеру и автоматические подсказки руководителю отдела продаж. Чтобы подключиться, вашей CRM
**не нужно ничего знать об устройстве Hunter AI** — достаточно отдавать данные в одном общем
формате, описанном ниже. Вместо того чтобы мы писали отдельный адаптер под каждую CRM, вы один
раз пишете небольшой «мостик» под наш стандарт — и подключение готово.

---

## 2. Авторизация

Поддерживаются два режима. Достаточно реализовать **любой один**; можно оба.

### Режим A — Pull (по умолчанию, самый простой)

Hunter AI периодически сам обращается к нескольким **read-only** эндпоинтам на вашей стороне.
Наши запросы вы аутентифицируете **статическим API-ключом**, который выдаёте нам один раз.
Ключ передаётся в заголовке `Authorization`:

```http
GET /hunter/leads?updated_since=2026-07-16T00:00:00Z HTTP/1.1
Host: crm.yourcompany.com
Authorization: Bearer hunter_live_9f2b7c8a1d4e...
Accept: application/json
```

Правила:
- Ключ генерируете вы, передаёте нам один раз (по защищённому каналу). Ротация в любой момент — просто сообщите новый.
- Только HTTPS.
- Запрос без валидного ключа — отклоняйте с `401 Unauthorized`.

### Режим B — Push через webhook (необязательно, для свежести данных)

Когда данные у вас меняются, вы можете сами присылать изменённую запись на URL, который мы вам
дадим, с общим секретом в заголовке (подпись тела запроса):

```http
POST https://ingest.hunterai.example/v1/webhook HTTP/1.1
Content-Type: application/json
X-Hunter-Signature: sha256=3a7f...   // HMAC-SHA256 тела запроса общим секретом
```

Push уменьшает задержку, но он **опциональный** — для старта достаточно режима Pull.

## 3. Обязательные сущности и форматы

Общие соглашения:
- **Все даты/время** — в формате ISO 8601 в UTC: `"2026-07-16T09:30:00Z"`.
- **Все `id`** — строки (число в виде строки тоже подойдёт): `"48213"`.
- **Телефоны** — по возможности в международном формате: `"+998901234567"`.
- Кодировка — UTF-8.

### 3a. Лид / Сделка

Основная сущность. Ключевой момент — поле `status.type`: пометьте каждый статус как `open`
(в работе), `won` (**финальный статус продажи**) или `lost` (**финальный статус потери**).
Так мы понимаем, что продажа состоялась, **не угадывая** по названию.

```json
{
  "id": "48213",
  "created_at": "2026-07-14T06:12:00Z",
  "updated_at": "2026-07-16T09:30:00Z",
  "responsible_employee_id": "1007",
  "contact_phone": "+998901234567",
  "status": {
    "id": "142",
    "name": "Оплачено",
    "type": "won"
  },
  "loss_reason": null,
  "payment": {
    "is_paid": true,
    "amount": 4500000,
    "currency": "UZS",
    "paid_at": "2026-07-16T09:28:00Z"
  }
}
```

Пример потерянного лида:

```json
{
  "id": "48090",
  "created_at": "2026-07-13T10:05:00Z",
  "updated_at": "2026-07-15T14:20:00Z",
  "responsible_employee_id": "1007",
  "contact_phone": "+998907654321",
  "status": { "id": "143", "name": "Закрыто", "type": "lost" },
  "loss_reason": { "id": "22", "name": "Неверный номер" },
  "payment": null
}
```

Обязательные поля:

| Поле | Тип | Описание |
|---|---|---|
| `id` | string | Уникальный идентификатор лида |
| `created_at` | datetime | Когда лид создан |
| `updated_at` | datetime | Последнее изменение (нужно для инкрементальной синхронизации) |
| `responsible_employee_id` | string | `id` ответственного сотрудника (см. 3c) |
| `contact_phone` | string \| null | Телефон контакта |
| `status.id` | string | `id` текущего статуса |
| `status.name` | string | Человекочитаемое название статуса |
| `status.type` | enum | `open` \| `won` \| `lost` — **обязательно** |
| `loss_reason` | object \| null | Причина потери (только если `status.type = lost`) |
| `payment` | object \| null | Факт оплаты (см. 3d, опционально) |

### 3b. Звонок

```json
{
  "id": "c-99120",
  "lead_id": "48213",
  "employee_id": "1007",
  "direction": "outbound",
  "started_at": "2026-07-16T08:41:12Z",
  "duration_seconds": 74,
  "answered": true
}
```

Обязательные поля:

| Поле | Тип | Описание |
|---|---|---|
| `id` | string | Уникальный идентификатор звонка |
| `lead_id` | string | К какому лиду относится звонок |
| `employee_id` | string | Кто звонил |
| `direction` | enum | `inbound` (входящий) \| `outbound` (исходящий) |
| `started_at` | datetime | Время начала звонка |
| `duration_seconds` | integer | Длительность разговора в секундах (0, если не ответили) |
| `answered` | boolean | Был ли дан ответ (соединение состоялось) |

### 3c. Сотрудник

```json
{
  "id": "1007",
  "name": "Ivan Petrov",
  "role": "sales"
}
```

| Поле | Тип | Описание |
|---|---|---|
| `id` | string | Уникальный идентификатор сотрудника |
| `name` | string | Имя (как показывать в отчётах) |
| `role` | string \| null | Роль, если есть (`sales`, `manager` и т.п.) — опционально |

### 3d. Оплата (опционально, но важно понимать)

Если ваша CRM **отдельно отслеживает факт оплаты** (а не только «сделка выиграна») — обязательно
передавайте это в поле `payment` у лида. Почему это критично, простыми словами:

> **«Выиграно» ≠ «деньги получены».** «Выиграно» означает, что в переговорах договорились о
> покупке. Но клиент может ещё не заплатить, оплатить частично или отменить. Если мы считаем
> выручку по статусу «выиграно», цифры будут завышены и решения — неточными. Когда есть реальный
> факт оплаты, метрики становятся честными: мы видим не «пообещали купить», а «деньги реально
> пришли».

```json
"payment": {
  "is_paid": true,
  "amount": 4500000,
  "currency": "UZS",
  "paid_at": "2026-07-16T09:28:00Z"
}
```

Если такого учёта у вас нет — передавайте `"payment": null`. Тогда мы работаем по статусу продажи
(это допустимо, просто менее точно). Ничего изобретать не нужно.

## 4. Требование к полноте данных

**Это обязательное правило, не рекомендация.**

Каждый ответ со списком (лиды, звонки, сотрудники) **должен содержать поле `is_complete`**
(boolean). Если по любой причине вы не смогли отдать все данные — сбой, лимит запросов, таймаут,
частичная выгрузка — верните `is_complete: false` и, по возможности, короткое пояснение в `note`.

```json
{
  "is_complete": false,
  "note": "rate limit reached, returned first 500 of ~1200 records",
  "leads": [ /* ... частичный список ... */ ]
}
```

**Почему это важно** (простыми словами): Hunter AI строит на этих данных управленческие решения —
кто из менеджеров отстаёт, куда уходят деньги, что подсказать руководителю. Если вы молча вернёте
**неполный** список без предупреждения, система примет его за полный и сделает **неверный вывод**
(например, решит, что менеджер не звонил, хотя данные о звонках просто не долетели). Явный флаг
`is_complete: false` — это сигнал «доверяй этим данным с осторожностью, часть отсутствует». Тогда
мы честно помечаем метрику как неполную, а не выдаём ложный вывод о человеке или отделе.

**Лучше сказать «данные неполные», чем молча отдать половину.**

## 6. Частые вопросы (FAQ)

**Q: У нас нет отдельного поля «причина потери». Что делать?**
Передавайте `"loss_reason": null`. Метрики по причинам потерь просто не будут строиться, но всё
остальное (конверсия, дозвон, скорость) работает. Если поле появится позже — просто начните его
отдавать, менять на нашей стороне ничего не нужно.

**Q: У нас не REST, а GraphQL / gRPC / выгрузка файлом. Подойдёт?**
Контракт — это **формат данных** (JSON-структуры выше), а не конкретный транспорт. Проще всего
отдать их обычным HTTPS-эндпоинтом (Pull) или webhook’ом (Push). Если у вас принципиально другой
способ — напишите нам, согласуем: главное, чтобы поля и их смысл совпадали с этим документом.

**Q: Как часто вы будете дёргать наш API?**
Точная частота **согласовывается индивидуально с каждым клиентом** — она зависит от лимитов вашего
API и от того, насколько свежими должны быть данные. Мы всегда передаём `updated_since`, чтобы вы
отдавали только изменённое с прошлого раза — полную базу каждый раз выгружать не нужно. Расскажите
о своих ограничениях по запросам заранее, и мы подберём комфортный для вашей стороны режим.

**Q: У нас нет отслеживания оплаты — только «сделка выиграна». Это проблема?**
Нет. Передавайте `"payment": null`, и мы работаем по статусу `type: "won"`. Просто помните: цифры
выручки будут отражать «договорились о покупке», а не «деньги пришли» (см. раздел 3d). Как только у
вас появится факт оплаты — начните отдавать `payment`, и метрики станут точнее автоматически.

**Q: Что считать `status.type = won` / `lost`, если статусов много?**
`won` — это финальный статус «продажа состоялась» (у вас может называться как угодно — «Оплачено»,
«Успешно», «Closed Won»). `lost` — финальный статус «сделка проиграна/закрыта без продажи». Все
промежуточные («Новый», «В работе», «Перезвонить» и т.п.) — `open`. Если финальных статусов
несколько — помечайте каждый соответствующим типом.

**Q: Нужно ли отдавать всю историю сразу?**
Для старта полезно отдать исторические данные (например, за последние 1–3 месяца) — одним первым
большим запросом без `updated_since`. Дальше идёт только инкремент по `updated_since`. Если история
большая — отдавайте страницами и на неполных страницах ставьте `is_complete: false`.
